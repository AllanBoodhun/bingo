- (decision by user) Win detection: server-side Postgres function/trigger fired on cell-check writes, writes winner(s) to Partie table. Binds: FR-10, FR-11, FR-12. Prevents: client-side divergence on who/when a winner is declared (race conditions, missed detections on lagging clients). Rule: clients never self-declare a win; they write their own cell state only and observe winner state via Realtime subscription on Partie.
- (decision by user) Hosting: Vercel for the static PWA frontend build; Supabase cloud project for backend. Binds: deployment/environments. Prevents: unnecessary infra ops for a solo, no-scale project. Rule: single environment (production) is sufficient; no multi-region or scaling infra.
- (decision by coach) Guest identity: Supabase anonymous auth, session persisted client-side (localStorage), survives reload/reconnect. Binds: FR-8, FR-15. Prevents: hand-rolled guest token/session scheme duplicating what Supabase Auth already provides. Rule: every Joueur (account or guest) has a Supabase auth identity (anonymous or persistent); no separate guest session mechanism.
- (decision by coach) Phrase text propagation: ADOPTED, derived directly from FR-3's tested consequence. Binds: Case/Phrase relationship. Prevents: denormalizing phrase text onto Case at distribution time, which would break live phrase-text corrections (FR-3) from reaching already-dealt grids. Rule: Case rows store a phrase_id foreign key + fixed grid position only, never a copy of phrase text; phrase text is always read live via join, so edits propagate automatically through existing realtime subscriptions.
- (decision by coach) Realtime mechanism: Supabase Realtime Postgres Changes (row-level change events) on Partie and Case tables, not a separate Broadcast channel. Binds: FR-3, FR-10 to FR-15. Prevents: maintaining two parallel realtime mechanisms (DB state + pub/sub messaging) for what is fundamentally one authoritative state store. Rule: all live UI updates (cell checked, phrase corrected, winner declared, game closed, transient toasts) are derived from Postgres Changes subscriptions on Partie/Case/Grille tables.
