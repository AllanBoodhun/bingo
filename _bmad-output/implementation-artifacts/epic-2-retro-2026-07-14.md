# Rétrospective — Epic 2 : Partie en temps réel

## Résumé de l'epic

- Stories complétées : 8/8 (2.1 à 2.8) — 100%
- FR couverts : FR-5 à FR-15, FR-18, FR-19, FR-20 (14 FR)
- Le MVP du projet (PRD §6.1, FR-1 à FR-20) est maintenant **entièrement couvert** par les Epics 1 et 2 — aucun Epic 3 n'existe dans `epics.md`.
- Chaque story a suivi le même cycle : implémentation → vérification manuelle (scripts Node/appels API directs, aucun framework de test — SM-C1) → revue de code adversariale (Blind Hunter, Edge Case Hunter, Acceptance Auditor) → correctifs → `done`.
- Toute la vérification s'est faite sur la stack Supabase **locale** — jamais encore en conditions réelles (Vercel + Supabase Cloud).

## Ce qui a bien fonctionné

- **La revue de code a systématiquement trouvé des bugs réels, jamais du bruit stylistique.** Sur les 8 stories, la revue a produit des patches non triviaux à chaque fois qu'un patch était nécessaire : verrou `for update` anti-TOCTOU sur le plafond de 6 joueurs (2.2), garde `.select()` contre un succès-fantôme filtré par RLS (2.3, réutilisée ensuite en 2.5/2.8), `vainqueurIdsRef` pour éviter une réouverture d'overlay sur événement redélivré (2.4), et surtout — le pattern le plus significatif de l'epic — **trois fuites Realtime inter-parties trouvées indépendamment par les trois couches de revue à la Story 2.5** (un créateur avec plusieurs parties actives via "Relancer" recevait les événements de la mauvaise partie). Ce dernier cas a directement informé la Story 2.6 : le risque était noté comme "à surveiller pour tout futur handler" avant même d'être retesté.
- **FR-20 (jeu solo) a été pensé en amont, pas retrofité.** Ajouté au PRD le 2026-07-04, *avant* le début de l'implémentation de l'Epic 2 — résultat : les Stories 2.1 à 2.7 ont toutes été conçues sans jamais supposer un minimum de joueurs, et la Story 2.8 s'est conclue par un diff vide côté application (audit + vérification bout-en-bout uniquement). C'est la preuve concrète qu'intégrer une exigence dans le PRD avant le développement évite complètement le travail de reprise a posteriori.
- **Le pattern fetch-then-subscribe (AD-10) a été posé une fois (Story 2.3) et étendu sans jamais être réinventé** (2.4 : vainqueurs, 2.5 : statut de partie, 2.6 : déclenchement automatique silencieux sur `online`/`visibilitychange`). Aucune des extensions n'a dû revenir sur la structure posée initialement.
- **Les décisions de design non couvertes par `DESIGN.md`** (overlay de vainqueur en 2.4, badge "Partie terminée" et bannière multi-parties en 2.5) ont été tranchées en restant strictement dans le vocabulaire de tokens existant (couleurs, rayons, espacements déjà nommés) et documentées inline avec leur raisonnement — pas d'improvisation visuelle hors système.
- **Le piège GRANT vs RLS** ("RLS filtre les lignes, GRANT filtre les colonnes") a été rappelé et correctement appliqué à chaque nouvelle table/colonne sensible (`cases.checked` en 2.3, `parties.statut` en 2.5) sans jamais être oublié, malgré sa nature contre-intuitive.

## Défis rencontrés

- **La classe de bug "Realtime scopé par RLS sans filtre explicite sur l'id de ressource" est le point faible récurrent de l'epic.** Elle est apparue dès que la RLS a cessé d'être 1:1 avec "ma seule ressource active" (multi-parties via "Relancer", ou un joueur authentifié qui rejoint la partie d'un autre). Trouvée et corrigée à chaque fois en revue, jamais en amont — un signal que ce pattern architectural (RLS large + canal non filtré) mérite d'être vérifié explicitement dès la conception d'un nouvel abonnement Realtime, plutôt que découvert après coup.
- **Aucune vérification en conditions réelles n'a encore eu lieu.** Toutes les 8 stories ont été vérifiées via scripts Node contre Supabase local et par lecture de code (E2E navigateur indisponible dans ce sandbox, limite documentée à chaque story). C'est un choix assumé (SM-C1, pas de sur-ingénierie de test pour un projet personnel) mais cela laisse un écart réel entre "vérifié" et "prouvé en usage" — d'autant que le succès du projet (SM-1) se définit justement comme un usage réel sans blocage technique.
- **L'activation de l'auth anonyme sur Supabase Cloud (production) n'a été confirmée qu'en local.** Le Debug Log de la Story 2.2 note explicitement que le flag `enable_anonymous_sign_ins` ne couvre que l'instance locale et que le provider doit être activé manuellement dans le Dashboard Supabase Cloud avant déploiement — un item resté ouvert depuis 2.2, jamais revérifié depuis.
